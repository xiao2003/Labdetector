# NeuroLab Hub 项目总览与使用手册

版本：1.0.0  
适用对象：实验室管理人员、实验操作人员、部署维护人员、演示与交付人员

## 一、项目定位

NeuroLab Hub 是面向实验室场景构建的多模态智能中枢系统。系统采用 **PC 中心端 + Raspberry Pi 边缘节点** 协同架构，统一承载：

- 多节点视频监控与关键帧上送
- 专家模型编排与风险提示
- 知识库问答与实验室语音交互
- 实验档案沉淀
- LLM 与视觉训练工作台

当前 `1.0.0` 基线的正式目标是形成一套：

- 可安装
- 可启动
- 可演示
- 可软著申报
- 可继续进行 Pi 实机联调

的软件版本。

## 二、系统架构

### 2.1 总体结构

系统收敛为：

- `PC`：中心控制端、GUI、知识库、专家模型执行、训练工作台、固定管家层调度
- `Pi`：轻前端节点，只负责唤醒、轻量 ASR、本地 TTS、本地停止播报、低频视觉前置检测、文本与关键帧上行
- AI 执行层：由用户在 GUI 中选择，用于知识问答生成、专家结果解释、视觉结果润色和开放式回答

### 2.2 固定管家层

当前版本已接入固定管家层架构，目标模型固定为：

- `Qwen2.5-1.5B-Instruct`
- 本地运行时：项目内置 `llama.cpp` Windows CPU runtime

固定管家层模型不暴露给用户，不在 GUI 中提供选择入口。它只承担：

- 意图识别
- 任务分发
- 多步编排
- 播报策略
- 事件摘要

执行层模型仍由用户在 GUI 中选择。

### 2.3 Pi 端角色边界

Pi 节点固定为“轻前端”，不承担本地大模型编排，不承担本地知识库问答，不承担本地专家选择。Pi 只负责：

- 唤醒
- 轻量 ASR
- 本地 TTS
- 本地停止播报
- 低频视觉前置检测
- 文本与关键帧上行

## 三、正式交付形态

### 3.1 Windows 端

当前正式交付形态为：

- `NeuroLab-Hub-Setup-v1.0.0.exe`
- 安装后主程序：`NeuroLab Hub.exe`
- 运行目录包含：
  - `_internal/`
  - `APP/`

安装包是正式交付形态。便携 `zip` 仅作为辅助发布形态。

### 3.2 Pi 端

当前正式交付形态为：

- `pi/` 目录整体复制
- 通过 `pc/一键配置树莓派.cmd` 完成首轮接入、Wi-Fi 配置、代码投递和后台自治安装

## 四、首次启动行为

### 4.1 GUI 首次启动原则

首次启动必须满足：

- GUI 先显示
- 固定管家层模型不得阻塞 GUI 主界面
- 首次启动后才后台执行固定管家层资产准备

### 4.2 后台准备内容

首次启动 GUI 就绪后，系统会后台执行：

1. 检查 `llama.cpp` runtime 是否就位
2. 检查固定 GGUF 是否已下载
3. 若缺失则后台下载模型
4. 下载完成后后台预热一次固定管家层模型
5. 将状态写入本地运行时状态文件

固定管家层状态固定为：

- `not_installed`
- `downloading`
- `download_failed`
- `warming_up`
- `ready`

固定管家层未就绪前，系统继续使用确定性规则链，不阻塞 GUI，也不导致空响应。

## 五、Windows 端使用方式

1. 运行 `NeuroLab-Hub-Setup-v1.0.0.exe` 完成安装。
2. 双击 `NeuroLab Hub.exe` 启动主程序。
3. 等待主界面出现。
4. 在 GUI 中选择执行层模型。
5. 导入公共知识与专家知识。
6. 导入专家资产。
7. 点击“开始监控”。
8. 等待 Pi 节点上线并进入闭环。

## 六、Pi 端接入方式

### 6.1 一键接入

推荐通过 Windows 端一键完成首轮接入：

1. 打开 `pc/pi_one_click_setup.json`
2. 至少填写：
   - `ssh.user`
   - `ssh.password`
3. 双击 `pc/一键配置树莓派.cmd`

脚本会自动：

- 发现可接入的 Pi
- 让 Pi 连接到与当前 PC 相同的 Wi-Fi
- 投递 `pi/` 目录
- 触发 Pi 后台自治安装
- 安装完成后自动启动 Pi 节点

### 6.2 Pi 常用命令

```bash
python3 pi_cli.py install-status
python3 pi_cli.py status
bash start_pi_node.sh
```

其中 `pi_cli.py status` 会显式输出：

- `voice.wake_word`
- `voice.wake_aliases`
- `architecture.node_role = light_frontend`
- `architecture.local_orchestration = false`

## 七、闭环能力

### 7.1 语音闭环

当前语音主链为：

1. Pi 本地离线识别音频为文本
2. 文本上行至 PC
3. PC 进入固定管家层编排
4. 再由执行层模型或专家执行链生成回答
5. 文本结果回传 Pi
6. Pi 本地播报

没有真实语音板时，仍可使用音频文件驱动 Pi 本地识别链完成闭环验证。

### 7.2 视频闭环

当前视频主链为：

1. Pi 低频前置检测
2. 仅在危险事件或用户主动请求时上传关键帧
3. PC 固定管家层统一编排
4. PC 调度专家模型与知识库
5. PC 返回结构化结果
6. Pi 完成播报和 ACK

### 7.3 播报策略

默认只保留两类自动播报：

- 用户主动询问
- 强告警

普通提醒默认只写事件摘要流，不自动播报。

## 八、当前正式边界

当前 `1.0.0` 已满足软件主链交付与演示要求，但仍存在明确边界：

- 多节点闭环目前以虚拟 Pi 和文件驱动路径验证为主
- 某些第三方语音扩展板存在兼容性风险，不纳入本版本正式通过项
- 真实麦克风阵列、真实扬声器阵列、实机集群长期联调仍需后续继续完成
