# NeuroLab Hub 项目总览与使用手册

版本：1.0.0  
适用对象：实验室管理人员、实验操作人员、部署维护人员、演示与交付人员

## 一、项目定位

NeuroLab Hub 是面向实验室场景构建的多模态智能中枢系统。系统采用 **PC 中心端 + Raspberry Pi 边缘节点** 协同架构，统一承载：

- 多节点视频监控与关键帧上送
- 专家模型编排与风险提示
- 知识库问答与实验室语音交互
- 实验档案沉淀
- LLM 与视觉训练工作台

当前 `1.0.0` 基线的正式目标是形成一套：

- 可安装
- 可启动
- 可演示
- 可软著申报
- 可继续进行 Pi 实机联调

的软件版本。

## 二、系统架构

### 2.1 总体结构

当前正式版本已收敛为 **6 层主链架构**，不是松散堆叠的功能集合。6 层从上到下分别是：

1. **交互与展示层**
    - Windows GUI
    - 首屏状态三态（系统已可用 / 后台准备中 / 后台准备失败）
    - 日志内单行进度条
    - 日志与事件流
    - 事件流筛选（全部 / 告警 / 调度 / 系统）
    - 最新高优事件常驻摘要卡片
    - 管家自治动作可视化痕迹
    - 知识导入业务结果反馈
    - 模型选择
    - 训练工作台
    - 文档入口
2. **会话与运行时层**
   - 启动链
   - 系统自检
   - 安装包首启
   - 固定管家层运行时状态管理
   - 内置模型校验与后台预热
3. **管家编排层**
   - `orchestrator`
   - 意图识别
   - 任务分发
   - 多步计划
   - 播报策略
   - 事件摘要
4. **执行与知识层**
   - 用户可选执行层模型
   - 知识库查询
   - 专家结果解释
   - 开放式回答
5. **通信与节点管理层**
   - `PC-Pi` 消息收发
   - 多节点状态管理
   - 事件回传
   - 结果回传
   - ACK
6. **Pi 轻前端边缘层**
   - 唤醒
   - 轻量 ASR
   - 本地 TTS
   - 本地停止播报
   - 低频视觉前置检测
   - 文本与关键帧上行

这 6 层里，只有第 4 层“执行与知识层”会被用户在 GUI 中切换模型。其余层属于固定系统主链。

### 2.2 固定管家层

当前版本已接入固定管家层架构，目标模型固定为：

- `Qwen3.5-0.8B`
- 本地运行时：项目内置 `llama.cpp` Windows CPU runtime

固定管家层模型不暴露给用户，不在 GUI 中提供选择入口。它只承担：

- 意图识别
- 任务分发
- 多步编排
- 播报策略
- 事件摘要

执行层模型仍由用户在 GUI 中选择。

### 2.3 执行层与管家层边界

当前版本严格采用双层模型架构：

- **固定管家层模型**
  - 固定为 `Qwen3.5-0.8B`
  - 本地运行时为项目内置 `llama.cpp`
  - 不暴露给用户
  - 只负责意图识别、任务分发、多步编排、播报策略和事件摘要
- **用户可选执行层模型**
  - 在 GUI 中选择
  - 只负责知识问答生成、专家结果解释、视觉结果润色和开放式回答

### 2.4 Pi 端角色边界

Pi 节点固定为“轻前端”，不承担本地大模型编排，不承担本地知识库问答，不承担本地专家选择。Pi 只负责：

- 唤醒
- 轻量 ASR
- 本地 TTS
- 本地停止播报
- 低频视觉前置检测
- 文本与关键帧上行

### 2.5 专家能力驱动调度

当前正式版本明确禁止把专家调度写成“固定场景 -> 固定专家”的硬编码链。系统调度依据固定为：

1. `expert_registry` 中的结构化能力元数据
2. 专家知识域是否存在且可用
3. 输入类型：
   - 语音命令
   - GUI 主动请求
   - 边缘事件
   - 本地模拟输入
4. 当前上下文：
   - 节点来源
   - 媒体类型
   - 是否需要关键帧
   - 是否属于主动询问或监控事件

固定管家层只能在已注册专家中做选择，允许近似匹配，但最终必须落回某个已注册 `expert_code`。专家知识域的作用是增强执行与解释，不是替代专家注册表的路由表。

## 三、正式交付形态

### 3.1 Windows 端

当前正式交付形态为：

- `NeuroLab-Hub-Setup-v1.0.0.exe`
- 安装后主程序：`NeuroLab Hub.exe`
- 运行目录包含：
  - `_internal/`
  - `APP/`

安装包是正式交付形态。便携 `zip` 仅作为辅助发布形态。

### 3.2 Pi 端

当前正式交付形态为：

- `pi/` 目录整体复制
- 通过 `pc/一键配置树莓派.cmd` 完成首轮接入、Wi-Fi 配置、代码投递和后台自治安装

## 四、首次启动行为

### 4.1 GUI 首次启动原则

首次启动必须满足：

- GUI 先显示
- 固定管家层模型不得阻塞 GUI 主界面
- 首次启动后才后台执行固定管家层资产准备

### 4.2 后台准备内容

首次启动 GUI 就绪后，系统会后台执行：

1. 检查 `llama.cpp` runtime 是否就位
2. 检查固定 GGUF 是否已随安装包就位
3. 若缺失则直接报为安装不完整
4. 模型就位后后台预热一次固定管家层模型
5. 将状态写入本地运行时状态文件

固定管家层状态固定为：

- `not_installed`
- `warming_up`
- `download_failed`
- `ready`

固定管家层未就绪前，系统继续使用确定性规则链，不阻塞 GUI，也不导致空响应。首次预热耗时较长时，状态会保持在 `warming_up`，而不是误报为失败。

### 4.3 首屏与事件流产品化口径

当前 GUI 正式口径已经收敛为面向用户的稳定表达，不再直接显示下载链、预热链等工程状态词。

首屏状态固定只显示三态：

- `系统已可用`
- `后台准备中`
- `后台准备失败（已回退规则链）`

主页面不再单独显示本机/节点进度卡片。所有进度统一写入系统事件流，并以单行进度条呈现。

节点进度沿用现有 WebSocket 文本协议回传，消息类型固定为：

- `PI_PROGRESS:{json}`

PC 只负责展示进度，不改变 Pi 作为轻前端的角色边界。

事件流顶部同时会常驻显示一张“最新高优事件”摘要卡片，用于提示：

- 强告警
- 节点异常
- 需要人工关注的自治动作失败

如果当前没有高优事件，卡片会显示稳定空态，而不是留空或闪烁。

日志区结构已经收敛为：

1. 最新高优事件卡片
2. 系统事件流摘要列表
3. 节点状态卡片
4. 事件详情区

节点区不再显示多张拥挤的小日志表，而是展示节点最近状态、最近摘要和当前任务进度。

## 五、应用内自治边界

当前版本允许的自治范围仅限应用内、运行期动作。固定管家层可以自动执行：

- 切换相关页面或视图
- 开始监控
- 停止监控
- 查询系统状态
- 调用已注册专家
- 查询知识库
- 请求关键帧上传
- 生成播报或静默策略
- 写事件摘要和会话日志

为保证自动动作可追踪，GUI 事件流会记录自治动作摘要，例如：

- `管家已执行动作: start_monitoring -> 监控已启动`
- `管家已执行动作: open_view -> 已切换到知识中心`
- `管家已执行动作: query_system_status -> 系统自检已发起`

这些痕迹只用于向用户解释系统为何自动切换页面、启动监控或触发专家，不改变业务行为本身。

当前版本明确不允许自动执行：

- 修改持久配置
- 修改用户持久模型设置
- 自动导入知识库或专家资产
- 自动发起训练或评估
- 自动执行 Windows、外部程序或文件系统级操作

## 六、知识导入反馈

当前 GUI 中，知识导入完成后的主反馈已经收敛成业务结果文案，而不是要求用户依赖目录或日志判断导入是否成功。

常驻反馈固定展示：

- `作用域`
- `新增文档数`
- `最近一次导入时间`

失败条目数和结构化记录数仍保留在详情对话框中，但不再作为主状态文案的核心信息。

## 七、Windows 端使用方式

1. 运行 `NeuroLab-Hub-Setup-v1.0.0.exe` 完成安装。
2. 双击 `NeuroLab Hub.exe` 启动主程序。
3. 等待主界面出现。
4. 在 GUI 中选择执行层模型。
5. 导入公共知识与专家知识。
6. 导入专家资产。
7. 点击“开始监控”。
8. 等待 Pi 节点上线并进入闭环。

## 八、Pi 端接入方式

### 8.1 一键接入

推荐通过 Windows 端一键完成首轮接入：

1. 打开 `pc/pi_one_click_setup.json`
2. 至少填写：
   - `ssh.user`
   - `ssh.password`
3. 双击 `pc/一键配置树莓派.cmd`

脚本会自动：

- 发现可接入的 Pi
- 让 Pi 连接到与当前 PC 相同的 Wi-Fi
- 投递 `pi/` 目录
- 触发 Pi 后台自治安装
- 安装完成后自动启动 Pi 节点

### 8.2 Pi 常用命令

```bash
python3 pi_cli.py install-status
python3 pi_cli.py status
bash start_pi_node.sh
```

其中 `pi_cli.py status` 会显式输出：

- `voice.wake_word`
- `voice.wake_aliases`
- `architecture.node_role = light_frontend`
- `architecture.local_orchestration = false`

## 九、闭环能力

### 9.1 语音闭环

当前语音主链为：

1. Pi 本地离线识别音频为文本
2. 文本上行至 PC
3. PC 进入固定管家层编排
4. 再由执行层模型或专家执行链生成回答
5. 文本结果回传 Pi
6. Pi 本地播报

没有真实语音板时，仍可使用音频文件驱动 Pi 本地识别链完成闭环验证。

### 9.2 视频闭环

当前视频主链为：

1. Pi 低频前置检测
2. 仅在危险事件或用户主动请求时上传关键帧
3. PC 固定管家层统一编排
4. PC 调度专家模型与知识库
5. PC 返回结构化结果
6. Pi 完成播报和 ACK

### 9.3 播报策略

默认只保留两类自动播报：

- 用户主动询问
- 强告警

普通提醒默认只写事件摘要流，不自动播报。

### 9.4 当前正式验证口径

当前版本的正式验证口径包括：

- 真实安装包首启 smoke
- GUI 首次启动不阻塞验证
- 固定管家层内置模型与后台预热状态验证
- 本地 PC 语音链进入 `orchestrator`
- 远端 Pi 语音文本进入同一编排主链
- Pi 边缘事件、`PI_EXPERT_EVENT`、`PI_YOLO_EVENT` 统一进入同一边缘事件编排入口
- 单节点与多节点虚拟闭环
- 音频文件驱动的 Pi 本地识别闭环

当前正式验收已经从安装包开始，覆盖：

1. 安装包交付层
2. 首次启动与运行时层
3. 交互与展示层
4. 管家编排层
5. 执行与知识层
6. 通信与节点管理层
7. Pi 轻前端边缘层

并串联完成 `PC-Pi` 语音闭环、`PC-Pi` 视频闭环和应用内自治关键场景验证。

## 十、当前正式边界

当前 `1.0.0` 已满足软件主链交付与演示要求，但仍存在明确边界：

- 多节点闭环目前以虚拟 Pi 和文件驱动路径验证为主
- 某些第三方语音扩展板存在兼容性风险，不纳入本版本正式通过项
- 真实麦克风阵列、真实扬声器阵列、实机集群长期联调仍需后续继续完成
