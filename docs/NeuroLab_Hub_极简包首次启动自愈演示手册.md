# NeuroLab Hub 极简包首次启动自愈演示手册

版本：1.0.0  
适用对象：演示人员、测试人员、交付人员

## 一、目标

本手册只覆盖 `NeuroLab_Hub_1.0.0_minimal.zip` 的首次启动演示，用于展示：

- 极简包不内置依赖与模型
- 系统首次启动后能够发现缺失项
- 系统能够按主链逐项自动补齐
- 补齐完成后能够再次自检并收敛到可用状态

## 二、极简包边界

当前极简包刻意移除了以下内容：

- `_internal`
- `Vosk` 离线语音资产
- 固定管家层 `llama.cpp` 组件
- 固定管家层 `Qwen3.5-0.8B.q4_k_m.gguf`

因此首次启动时，系统出现“发现缺失 -> 自动补齐 -> 自动复检”是预期行为，不是异常。

## 三、演示前准备

### 3.1 环境要求

- Windows 主机可联网
- 当前目录可写
- 允许程序下载依赖和模型资产
- 不手动提前拷贝 `Vosk`、固定管家层组件或 GGUF

### 3.2 演示文件

- 极简包：`NeuroLab_Hub_1.0.0_minimal.zip`
- 当前 Release：`v1.0.0`

### 3.3 演示禁忌

- 不要手动修改 `config.ini`
- 不要手动创建模型目录或语音资产目录
- 不要手动把完整包里的资产拷入极简包目录
- 不要在补齐过程中强制关闭程序

## 四、首次启动演示脚本

### 4.1 开场说明

建议开场口径：

“当前演示使用的是极简包。它不内置语音资产和固定管家层模型，系统会在首次启动时自行发现缺失项、自动补齐，并在补齐完成后再次自检，最终收敛到可用状态。”

### 4.2 实际操作步骤

1. 解压 `NeuroLab_Hub_1.0.0_minimal.zip`
2. 启动主程序
3. 观察首次启动加载界面或主界面中的状态变化
4. 等待系统进入自检
5. 观察系统逐项补齐缺失依赖、语音资产与固定管家层组件
6. 等待系统自动再次自检
7. 进入主界面后确认系统状态已经收敛

### 4.3 现场讲解重点

建议按以下顺序讲解：

1. “当前是极简包，不包含完整依赖与模型。”
2. “系统已经识别到缺失项，并开始逐项补齐。”
3. “补齐完成后，系统不会停留在中间状态，而是会自动再次自检。”
4. “最终系统状态收敛后，才能进入后续语音、知识和监控主链。”

## 五、预期可见现象

### 5.1 首次启动阶段

预期会看到：

- 系统进入启动检查
- 缺失项被识别
- 进度逐步推进
- 补齐动作自动发生

### 5.2 补齐完成后

预期会看到：

- 系统自动进入再次自检
- 固定管家层状态从缺失/准备中逐步恢复
- 主界面最终进入可用状态

### 5.3 演示成功判定

满足以下条件即可判定“首次启动自愈演示通过”：

- 至少出现一次明确的缺失识别
- 至少完成一次自动补齐
- 自动补齐后触发再次自检
- 最终系统进入可用状态或明确收敛到可继续操作的状态

## 六、演示后的继续动作

首次启动自愈演示完成后，推荐继续演示以下主链：

1. 管家语音处理
2. 专家知识导入
3. 专家语音问答
4. 固定管家层调度专家并播报结果
5. 常驻专家模型识别危险并触发处理链

说明：

- 如果首次启动补齐还未完全结束，不建议立即进入语音和监控演示
- 应先确认系统已经完成自动复检并稳定

## 七、失败时的判断方式

### 7.1 属于正常等待的情况

- 下载仍在进行
- 固定管家层仍在准备中
- 自动复检尚未结束

### 7.2 属于需要重新处理的情况

- 长时间没有任何进度变化
- 补齐动作中断且没有再次自检
- 启动后无法再次进入主界面

### 7.3 处理原则

- 优先等待系统完成当前补齐动作
- 若确认网络中断或下载失败，再重新启动程序触发自检链
- 不手动篡改目录内容，不跳过正式自检链

## 八、推荐演示话术

### 8.1 自愈阶段

“这里使用的是不带模型和依赖的极简包。系统会先发现缺失项，再自动补齐，补齐后还会自动复检，确保不是只完成下载，而是真正进入可用状态。”

### 8.2 进入主链阶段

“当自检和自愈收敛完成后，后面的管家语音、专家知识和监控能力才会进入正式主链。”

## 九、结论

极简包的价值不在于直接开箱即用，而在于验证系统是否具备真正的首次启动自愈能力。演示时应把重点放在：

- 能否识别缺失
- 能否自动补齐
- 能否自动复检
- 能否最终收敛
