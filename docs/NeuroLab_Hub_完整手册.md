## 2. 项目概览

NeuroLab Hub 是一套面向实验室场景的多模态智能中枢系统，采用“PC 中心端 + Pi 边缘节点 + 专家模型 + 知识库 + 本地/云端模型服务 + 训练工作台”的整体架构，为实验室安全监控、危化品识别、语音交互、知识沉淀和模型迭代提供统一的软件化平台。

项目目标不是单纯做一个问答界面，而是构建一条完整闭环：

1. Pi 侧采集实验现场视频、关键帧和语音。
2. PC 侧下发策略、接收关键事件、调度专家与知识库。
3. 统一多模态模型根据专家结果和知识库做专业解释。
4. 结果回传 Pi 播报，同时写入实验档案和知识体系。
5. 数据再回灌到训练工作台，支持后续模型迭代。

## 3. 当前交付基线

### 3.1 当前版本重点

当前 3.0.9 版本已经聚焦成最小可交付闭环，优先保证以下能力：

- 实验室着装安全闭环
- 危化品识别与播报闭环
- 语音记录、整轮知识提取与知识库入库闭环
- PC-Pi 单节点与多节点闭环
- GUI 可视化监控、模型配置、知识库导入、训练入口

### 3.2 当前交付原则

当前版本遵循以下交付原则：

1. 代码仓库不再打包大模型。
2. 语音模型与运行依赖在启动后的自检阶段自动补齐。
3. Ollama 模型在首次选择时自动拉取。
4. EXE 只保留轻量入口，控制总体积维持在几十 MB 到一百多 MB 量级。

### 3.3 当前默认聚焦专家

当前用于 PC-Pi 闭环的专家已收敛为：

- `safety.ppe_expert`
- `safety.chem_safety_expert`

语音记录链路仍保留完整归档与知识提取逻辑，其他专家默认不进入最小闭环策略下发。

## 4. 系统总体架构

### 4.1 架构分层

系统可拆成五层：

1. **交互层**  
   Windows GUI、启动页、工作台、系统事件流、节点日志、知识库导入窗口、模型配置窗口。

2. **调度层**  
   PC 中心端运行时、会话管理、GUI 状态同步、节点扫描与 WebSocket 通信。

3. **感知层**  
   专家模型、OCR、检测、姿态、安全规则、边缘侧裁剪策略。

4. **知识增强层**  
   公共知识库、专家知识域、统一多模态 LLM 解释层。

5. **训练与交付层**  
   训练工作台、打包、测试、发布。

### 4.2 PC 与 Pi 的职责分工

**PC 中心端负责：**

- 节点发现与会话管理
- 专家模型调度
- 知识库检索与增强
- 多模态 LLM 解释
- GUI 展示与档案沉淀
- 训练工作台与发布管理

**Pi 边缘节点负责：**

- 摄像头接入
- 轻量策略判断
- 关键帧裁剪与上报
- 语音采集
- 回传播报

### 4.3 当前闭环设计

当前闭环不走“Pi 端直接全量推理”，而是采用：

1. Pi 接收 PC 下发的专家裁剪策略。
2. Pi 平时发低频预览，关键时刻只发裁剪关键帧。
3. PC 按 Pi 回传的 `expert_code / policy_name / policy_action` 调用对应专家。
4. 专家输出结构化结果。
5. 若该专家启用知识增强，则结合专家知识域与统一多模态 LLM 生成最终专业解释。
6. PC 将结论和播报文本回传给 Pi。
7. Pi 播报结果。

## 6. 运行环境与部署前提

### 6.1 PC 推荐环境

- Windows 10/11 64 位
- Python 3.11
- 16GB 以上内存
- NVIDIA GPU（推荐）
- 已安装 Git
- 已安装 Ollama（或允许自检自动安装）

### 6.2 Pi 推荐环境

- Raspberry Pi 4/5
- Raspberry Pi OS
- 摄像头、麦克风、扬声器
- 与 PC 同局域网

### 6.3 本地模型与性能建议

当前 GUI 中默认保留的 Ollama 模型有：

- `gemma3:4b`
- `llama3.2-vision:11b`
- `qwen3.5:4b`
- `qwen3.5:9b`
- `qwen3.5:27b`
- `qwen3.5:35b`

推荐建议：

| 模型 | 推荐用途 | 适用机器 |
| --- | --- | --- |
| `gemma3:4b` | 最快的本地默认模型 | 一般笔记本 / 普通 GPU |
| `qwen3.5:4b` | 国产轻量问答主力 | 普通 GPU |
| `qwen3.5:9b` | 国产中档综合模型 | 4060 及以上 |
| `llama3.2-vision:11b` | 图像理解补充 | 中高配 GPU |
| `qwen3.5:27b` | 高质量中文推理 | 高显存设备 |
| `qwen3.5:35b` | 高质量增强档 | 高显存设备 |

当前为了提升一般性能电脑上的可用性，`qwen3.5` 已自动套用轻量推理预设，并关闭 `thinking` 输出，优先直接返回最终答案。

## 7. 安装、首次启动与自检

### 7.1 轻量交付策略

当前交付包只保留轻量入口 EXE。第一次运行后：

1. 入口先检查 Python 环境。
2. 主程序启动后运行自检。
3. 自检自动检查并补齐：
   - 核心 Python 依赖
   - Ollama 运行时
   - SenseVoice 模型
   - Vosk 模型

### 7.2 首次启动体验

启动页采用固定阶段式引导，用户能看到：

- 环境检查
- 启动自检
- 配置加载
- 界面就绪
- `99% 正在启动工作台界面`

### 7.3 首次使用推荐顺序

1. 启动主程序。
2. 等待自检完成。
3. 配置 AI 模型。
4. 导入公共知识库与专家知识库。
5. 先做单机验证。
6. 再做 Pi 节点接入和闭环验证。

## 8. GUI 操作说明

### 8.1 顶部主导航

GUI 顶部保留这些常用入口：

- 界面侧栏
- 系统自检
- 专家中心
- 知识中心
- 模型配置
- 训练中心
- 使用手册
- 关于系统

### 8.2 左侧运行配置

当前左侧运行配置重点项为：

- `AI模型`
- `模型选择`
- `运行模式`
- `预期节点数`
- `启动监控 / 停止监控`

#### AI模型

当前左侧 `AI模型` 保留：

- `Ollama`
- `通义千问（需在模型配置页面添加 API Key）`
- `DeepSeek（需在模型配置页面添加 API Key）`
- `Kimi（需在模型配置页面添加 API Key）`
- `本地模型框架`

#### 模型选择

- `Ollama` 默认提供官方本地模型候选
- 云端模型默认不预塞候选
- 云端模型需在“模型配置”中填写 API 与“模型命名”
- 保存后会自动加入对应 `AI模型` 的 `模型选择`
- 左侧可用统一 `删除` 按钮移除已添加的自定义模型

### 8.3 右侧主工作区

右侧主工作区会随运行模式动态变化：

- 待机 / 单机模式：主系统事件流优先
- 多节点 WebSocket 模式：系统事件流 + 节点日志卡片

### 8.4 日志区布局

当前日志区设计遵循：

1. 主系统事件流保留系统级消息
2. 节点上行语音 / 下行播报 / 节点通信单独显示在各自节点卡片
3. 节点卡片按每行两个平铺
4. 整个日志区支持整体滚动，便于查看多节点

## 10. 语音交互与状态机

### 10.1 语音链路

当前语音交互使用本地状态机统一调度，分成：

- 唤醒
- 控制
- 路由
- 问答
- 专家唤醒
- 知识提取

### 10.2 当前控制优先级

控制词具有最高优先级，本地直接执行，不再先送大模型：

- `停止播报`
- `不用了`
- `结束语音`

### 10.3 当前记录逻辑

当前整轮语音会：

1. 完整保留原始对话归档
2. 在结束本轮时只从“用户口述”中提取有效知识
3. 不把专家模型结果直接写入知识库

## 12. PC-Pi 闭环说明

### 12.1 单节点闭环

单节点场景中：

1. Pi 只发送低频弱预览帧
2. 关键事件时发送裁剪关键帧和策略元信息
3. PC 调用对应专家并研判
4. 结果回传 Pi
5. Pi 进行语音播报

### 12.2 多节点闭环

多节点场景中：

1. PC 可同时维护多个节点会话
2. 每个节点分别保留自己的日志卡片
3. 系统事件流只保留系统级消息
4. 每个节点的上行语音、下行播报和通信事件独立展示

### 12.3 当前已验证的闭环能力

已通过：

- 单节点闭环
- 多节点闭环
- 语音记录闭环
- 危化品识别与播报闭环
- PPE 与知识增强闭环

## 19. 结语

当前版本已经形成一条可交付、可运行、可测试、可发布的最小闭环体系。后续如果继续迭代，建议坚持以下原则：

1. 先保证闭环，不先追求模型种类膨胀。
2. 先保证专家结果与知识库解释一致，再扩展新专家。
3. 先保证交付轻量与自恢复能力，再扩展大型本地模型。
4. 所有新能力都必须先进入 `test/pc` 与 `test/pi`，再进入交付包。

这份完整手册后续应作为文档总入口持续维护，避免说明分散、版本脱节和交付口径不一致。
