# NeuroLab Hub —— 基于可编排专家模型的实验室多模态智能中枢（正式版 1.0.0）

[![Python 3.11+](https://img.shields.io/badge/python-3.11+-blue.svg)](https://www.python.org/downloads/)
[![Windows Desktop](https://img.shields.io/badge/Delivery-Windows%20Desktop-1f6feb)](docs/NeuroLab_Hub_用户手册.md)
[![Raspberry Pi](https://img.shields.io/badge/Edge-Raspberry%20Pi-green)](docs/NeuroLab_Hub_软件说明书.md)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)

NeuroLab Hub 是面向科研实验室场景构建的分布式多模态智能系统。系统以 **PC 中心端 + Raspberry Pi 边缘节点** 协同为基础，将实验现场监控、专家模型编排、知识库检索增强、语音交互、实验档案沉淀以及训练工作台整合到同一套运行体系中，服务于实验室安全巡检、危化品识别、实验过程记录、知识沉淀和模型迭代。

当前正式版聚焦三条最小可用闭环：

* 实验室着装安全闭环
* 危化品识别与播报闭环
* 语音记录、整轮提取与知识入库闭环

## 核心特性

* **中心-分布式协同架构**  
  `pc/` 与 `pi/` 两端职责清晰分离。PC 中心端负责节点管理、策略下发、专家分析、知识检索、统一解释与结果归档；Pi 边缘端负责视频采集、边缘触发、语音输入、关键帧裁剪与结果播报。
* **专家模型可编排**  
  系统通过统一专家注册表、专家管理器与闭环协议组织危化品识别、PPE 穿戴、设备 OCR、火焰烟雾、液体洒漏、手部姿态、实验问答、微纳流体等专家能力，并支持按交付目标裁剪闭环策略。
* **知识库分层增强**  
  支持 `common` 公共知识库与 `expert.*` 专家专属知识库并存。传统 CV 专家先给出结构化结果，再结合专家知识域交给统一多模态 LLM 做专业解释。
* **本地优先的语音状态机**  
  语音链路采用“唤醒、控制、路由、问答、专家、知识提取”统一调度，本地支持停止播报、结束语音、记录整轮口述、危化品专家唤醒和 Pi 回传播报。
* **训练与回灌闭环**  
  内置 LLM 微调工作台与 YOLO 训练工作台，支持数据导入、训练工作区构建、训练结果注册与运行目录回灌。
* **轻量交付策略**  
  EXE 保持轻量入口体积，Python 运行环境、Ollama、本地语音模型和 Ollama 模型均在首次启动 / 自检 / 首次选择阶段按需补齐，不随源码仓库或发布包强行分发。

## 项目定位

NeuroLab Hub 面向以下典型问题：

* 实验现场监控与记录依赖人工值守，缺少连续、结构化、可追溯的采集与分析能力。
* 实验室风险识别依赖个人经验，缺少稳定规则、统一模型与结果闭环机制。
* 实验语音问答、风险提示和实验记录分散在不同工具中，缺少统一交互入口。
* SOP、设备手册、危化品规范、FAQ 和历史记录难以沉淀为可检索知识库。
* 训练数据、模型产物与运行系统割裂，缺少从数据到训练、再到部署回灌的完整闭环。

项目以软件系统化交付为目标，形成围绕“监控、分析、问答、归档、训练、回灌”的实验室智能中枢。

## 总体架构

### 1. 端到端中心-分布式架构

```text
┌────────────────────────────────────────────────────────────────────────────┐
│                            PC 中心端（控制与研判）                         │
│  ├─ GUI 工作台 / 运行配置 / 模型配置                                       │
│  ├─ 多节点管理 / WebSocket 会话 / 策略下发                                 │
│  ├─ 专家模型调度 / 统一多模态 LLM 解释层                                   │
│  ├─ 知识库检索增强 / 语音状态机 / 实验档案                                 │
│  └─ 训练工作台 / 模型注册 / 结果回灌                                       │
└────────────────────────────────────────────────────────────────────────────┘
                     ▲                              │
                     │ 结果回传 / TTS / 策略同步     │ 事件 / 关键帧 / 语音命令
                     │                              ▼
┌────────────────────────────────────────────────────────────────────────────┐
│                          Pi 边缘节点（采集与执行）                         │
│  ├─ 摄像头采集 / 弱预览帧                                                  │
│  ├─ 按策略触发关键裁剪帧                                                   │
│  ├─ 边缘语音输入 / 本地播报                                                 │
│  ├─ 事件元信息 / 专家策略标记                                               │
│  └─ WebSocket 节点通信                                                      │
└────────────────────────────────────────────────────────────────────────────┘
```

### 2. 核心链路分工

1. **Pi 端负责采集与预处理**  
   采集视频、语音和节点状态；平时发送弱预览帧，关键时刻按策略发送裁剪关键帧。
2. **PC 端负责调度与分析**  
   接收事件后调用对应专家、知识库和统一解释层完成风险判断。
3. **结果回传 Pi 端执行**  
   PC 将文字结论、播报文本和控制指令回传给 Pi，由 Pi 负责本地播报与现场执行。

### 3. 当前正式版聚焦闭环

正式版 1.0.0 默认只让以下能力进入 PC-Pi 闭环：

* `safety.ppe_expert`
* `safety.chem_safety_expert`
* 语音记录、整轮知识提取与入库

其他专家仍保留在系统中，但不默认进入交付级闭环策略下发。

## 从启动到闭环的运行过程

### 1. 首次启动

1. 启动轻量入口程序。
2. 入口检测 `Python 3.11` 是否存在；若缺失，先安装 Python。
3. 主程序启动后进入固定阶段启动页，完成环境检查、启动自检、配置加载与界面初始化。

### 2. 自检补齐运行环境

1. 检查本地依赖、训练环境、GPU、语音运行时和知识库目录。
2. 如缺少 `Ollama`，自检阶段自动尝试下载安装。
3. 如缺少本地语音模型（`SenseVoice / Vosk`），自检阶段自动下载补齐。
4. 不再将语音模型或本地大模型打进 EXE / 仓库。

### 3. 模型选择与模型配置

1. 在左侧 `AI模型` 选择模型来源：
   * `Ollama`
   * `通义千问（需在模型配置页面添加 API Key）`
   * `DeepSeek（需在模型配置页面添加 API Key）`
   * `Kimi（需在模型配置页面添加 API Key）`
   * `本地模型框架`
2. 在右上角 `模型配置` 中录入云端模型的 API Key、模型 ID 和显示名称。
3. 保存后，该模型名称会加入左侧对应 `AI模型` 的 `模型选择` 下拉框。
4. 选择 `Ollama` 模型时，如本机缺失该模型，会在首次使用时自动拉取。

### 4. 中心端与边缘端建立闭环

1. PC 端选择运行模式：
   * 单机摄像头模式
   * 多节点 WebSocket 模式
2. 多节点模式下，PC 扫描并连接 Pi 节点。
3. PC 向 Pi 下发当前聚焦闭环的专家策略。
4. Pi 依据策略进行边缘触发和关键帧裁剪。

### 5. 事件分析与统一解释

1. Pi 将关键帧、事件类型、策略信息和语音命令回传给 PC。
2. PC 调用对应专家模型完成结构化识别。
3. 若该专家启用知识增强解释层，则进一步检索对应知识库。
4. 统一多模态 LLM 根据“图像 + 专家结果 + 知识上下文”输出专业研判。

### 6. 结果回传与沉淀

1. PC 将最终提示、建议动作和播报文本回传给 Pi。
2. Pi 本地语音播报结果。
3. PC 同时把会话、事件、语音轮次和关键帧写入实验档案。
4. 整轮语音结束后，仅从用户口述中提取有效知识写入知识库。

## 功能模块说明

### 1. 桌面中心端

桌面中心端主要由 `pc/desktop_app.py` 驱动，当前包括：

* 会话状态展示、顶部工作台入口与系统概览
* 单机监控与多节点监控
* 运行配置、模型选择、运行模式切换
* 知识库管理、专家中心、模型配置
* 启动自检、使用手册、关于系统
* 训练中心与实验档案中心入口

### 2. 多节点扫描与通信模块

相关模块主要位于 `pc/communication/` 与 `pi/pisend_receive.py`，支持：

* 局域网节点扫描
* WebSocket 多节点管理
* 中心端策略下发
* 关键帧与事件回传
* 专家分析结果回传 Pi 端
* 节点状态同步、结果闭环与运行审计

### 3. 启动自检与运行检查

PC 与 Pi 两端均提供自检与环境检查能力：

* PC 端在 `pc/webui/runtime.py` 中完成运行环境、模型服务、知识库目录和训练依赖检查。
* Pi 端通过 `pi/pi_cli.py` 和相关运行流程完成命令、依赖和节点环境检查。
* 发布流程要求执行编译检查、功能测试、入口烟测和文档一致性检查。

### 4. 实验档案中心

实验档案能力位于 `pc/core/experiment_archive.py`，用于沉淀：

* 会话元数据
* 关键帧文件
* 事件类型与事件内容
* 风险提示与专家分析结果
* 语音交互内容
* 时间线记录与会话关闭信息

### 5. 知识库系统

知识库能力位于 `pc/knowledge_base/`，包括：

* `rag_engine.py`：向量检索与知识管理主入口
* `structured_kb.py`：结构化 SQLite 知识库
* `media_ingestion.py`：图片、音频、视频资料导入与摘要处理

当前知识库支持：

* `common` 公共底座知识库
* `expert.*` 专家专属知识库
* 文本、Markdown、图片、音频、视频导入
* 结构化记录与向量索引并行组织

### 6. 专家模型体系

专家注册表位于 `pc/core/expert_registry.py`，专家加载与调度由 `pc/core/expert_manager.py` 负责。当前已接入的核心专家包括：

#### 实验室安全规范相关专家
* `safety.chem_safety_expert`
* `safety.equipment_operation_expert`
* `safety.flame_fire_expert`
* `safety.general_safety_expert`
* `safety.hand_pose_expert`
* `safety.integrated_lab_safety_expert`
* `safety.ppe_expert`
* `safety.spill_detection_expert`

#### 实验知识与问答相关专家
* `lab_qa_expert`
* `equipment_ocr_expert`

#### AI for Science 研究辅助相关专家
* `nanofluidics.microfluidic_contact_angle_expert`
* `nanofluidics.nanofluidics_multimodel_expert`

### 7. AI 模型与模型配置

#### 7.1 模型来源

当前 GUI 左侧 `AI模型` 采用“本地优先、云端补充”的方式：

* `Ollama`
* `通义千问（需在模型配置页面添加 API Key）`
* `DeepSeek（需在模型配置页面添加 API Key）`
* `Kimi（需在模型配置页面添加 API Key）`
* `本地模型框架`

说明：

* `Ollama` 置顶，作为正式版默认的本地闭环方案。
* 云端模型统一通过 GUI 的 `模型配置` 页面录入，不预塞默认模型。
* `模型配置` 保存后，模型名称会回写到左侧对应 `AI模型` 的 `模型选择` 下拉框。
* 左侧 `模型选择` 支持统一删除当前自定义模型。

#### 7.2 Ollama 默认模型

正式版默认保留以下本地模型入口：

* `gemma3:4b`
* `llama3.2-vision:11b`
* `qwen3.5:4b`
* `qwen3.5:9b`
* `qwen3.5:27b`
* `qwen3.5:35b`

说明：

* 本地 Ollama 模型不随项目或 EXE 打包。
* 目标电脑首次选择某个模型时，由本机 Ollama 自动拉取。
* 若目标电脑缺少 Ollama，系统会在“系统自检”阶段自动尝试安装。

#### 7.3 模型性能建议（按用户机器配置选择）

以下建议采用“轻量 / 主力 / 高质量”分级，便于根据机器配置直接选择：

| 使用定位 | 推荐模型 | 适用设备 | 主要用途 | 说明 |
| --- | --- | --- | --- | --- |
| 轻量启动 | `gemma3:4b` | 普通 CPU / 入门 GPU | GUI 验证、基础问答、低负载调试 | 启动快、占用低，适合作为首轮部署模型 |
| 轻量国产 | `qwen3.5:4b` | 轻量笔记本 / 低显存设备 | 基础实验问答、知识检索增强 | 适合希望使用国产模型但机器性能一般的场景 |
| 日常主力 | `qwen3.5:9b` | RTX 4060 及以上 | 日常实验室问答、知识解释、单节点闭环 | 推荐作为国产主力模型 |
| 视觉解释 | `llama3.2-vision:11b` | 中高端 GPU | 图像理解、标签解释、视觉问答 | 适合视觉任务较多的场景 |
| 高质量研判 | `qwen3.5:27b` | 高显存 GPU / 工作站 | 多节点研判、复杂专业解释 | 适合高质量分析，不适合日常默认 |
| 旗舰离线 | `qwen3.5:35b` | 高端工作站 / 充足显存 | 复杂长文本、多轮专业解释 | 作为旗舰模型使用，不建议普通电脑默认开启 |

#### 7.4 普通性能电脑推荐

若使用 `RTX 4060 Laptop GPU` 或类似级别机器，推荐按以下顺序选择：

1. `gemma3:4b`
2. `qwen3.5:4b`
3. `qwen3.5:9b`

不建议将 `qwen3.5:27b` 和 `qwen3.5:35b` 设为日常默认模型。

#### 7.5 本地推理优化策略

系统会对 `Qwen3.5` 系列自动套用轻量推理预设：

* `qwen3.5:4b`：`4096` 上下文、`192` 最大生成长度
* `qwen3.5:9b`：`3072` 上下文、`160` 最大生成长度
* `qwen3.5:27b`：`2048` 上下文、`128` 最大生成长度
* `qwen3.5:35b`：`2048` 上下文、`96` 最大生成长度

同时默认关闭 `thinking` 轨迹输出，优先返回最终中文答案，减少普通电脑上的等待时间。

### 8. 训练工作台

训练相关模块位于 `pc/training/`，当前支持：

* 导入 LLM 微调语料
* 导入 YOLO 检测训练数据
* 构建训练工作区
* 检查训练依赖与本地运行时
* 启动训练任务
* 记录训练日志
* 自动注册训练结果并回灌运行目录

## 硬件准备与部署清单

### 1. 中心计算节点（PC 端）

负责多节点管理、专家分析、知识检索、语音理解、桌面交互与训练控制。

* **操作系统**：Windows 10 / Windows 11，64 位
* **算力设备（GPU）**：建议 NVIDIA RTX 3090 / 4090 / 5090 或同等级设备
* **内存（RAM）**：建议 32GB 及以上
* **存储空间**：建议预留 50GB 以上
* **外设交互**：如需本机语音交互，建议配置麦克风与音频输出设备

### 2. 边缘节点（Raspberry Pi）

负责实验位点的视频采集、事件触发、语音输入与边缘执行。

* **主控板**：Raspberry Pi 4B / 5 或同等级 ARM Linux 设备
* **摄像头**：CSI 或 USB 摄像头
* **音频设备**：按场景配置麦克风与扬声器
* **存储与供电**：建议 32GB 以上存储，并使用稳定电源

### 3. 网络环境要求

* **网络拓扑**：PC 与全部 Pi 节点处于同一局域网
* **链路稳定性**：建议千兆有线网络或高质量 Wi-Fi
* **模型服务连通性**：本地或云端模型服务需能被中心端访问

## 上手指南

### 0. 获取源码

```bash
git clone https://github.com/xiao2003/Labdetector.git
cd Labdetector
```

### 1. 中心端部署（PC）

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
python launcher.py
```

### 2. 边缘端部署（Raspberry Pi）

```bash
cd pi
./start_pi_node.sh start --auto-install-deps
```

仅查看命令帮助时可执行：

```bash
python pi_cli.py --help
```

### 3. 知识库与专家能力准备

1. 导入 `common` 公共知识库资料
2. 按需导入对应 `expert.*` 专家知识库资料
3. 为需要模型文件的专家导入模型资产或样本文件
4. 在桌面端完成知识库、专家和模型配置检查

### 4. 训练工作台

系统提供两个训练入口：

* `launcher.py --annotation-panel`：标注与视觉训练入口
* GUI 中的 `训练中心`：LLM / YOLO 训练管理入口

### 5. 自动化测试与发布验收

正式版 1.0.0 已完成：

* GUI 静默回归
* 模块总回归（21 pass / 0 warn / 0 fail）
* 视觉训练冒烟
* PPE / 危化品 / 语音记录聚焦闭环
* PC-Pi 单节点闭环
* PC-Pi 多节点闭环与压力测试

推荐测试入口：

```powershell
python test/pc/auto_module_test.py
python test/pc/run_gui_silent_test.py
python test/pc/test_visual_training_pipeline.py
python test/pi/test_pc_pi_integrated_bridge.py
python test/pi/test_pc_pi_llm_interpreter_closed_loop.py
```
