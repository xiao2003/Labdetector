NeuroLab Hub 是面向科研实验室场景构建的分布式多模态智能系统。系统采用 `PC` 中心端与 `Raspberry Pi` 边缘节点协同架构，覆盖实验现场监控、专家模型编排、知识导入、训练工作台、语音交互、视觉闭环和实验档案沉淀。

当前 `1.0.0` 正式基线已经收敛为：

1. Windows 端以 `Setup.exe` 安装包作为正式交付形态。
2. `Pi` 端复制 `pi/` 目录即可启动边缘节点。
3. GUI 先显示，固定管家层模型直接使用安装包内置 GGUF 在后台预热。
4. 用户在 GUI 中只选择执行层模型，固定管家层模型不暴露。
5. `PC-Pi` 语音/视频闭环已统一收敛到单一编排主链。
6. 当前系统架构已收敛为 6 层主链，职责边界明确。
7. 专家调度以专家元数据、知识域可用性和输入上下文为事实来源，不再依赖固定场景硬编码。

## 当前正式交付形态

- 安装包：`NeuroLab-Hub-Setup-v1.0.0.exe`
- 便携包：`NeuroLab_Hub_1.0.0.zip`
- 复验包：`NeuroLab_Hub_1.0.0_fresh_validation.zip`
- Pi：`pi/` 目录整体复制

普通用户优先使用安装包。便携包用于需要解压即用的场景。

## 首次启动说明

首次启动时：

- 主界面会先显示
- 固定管家层模型不会阻塞 GUI
- 系统会在后台检查 `llama.cpp` runtime、校验安装包内置 GGUF、后台预热模型
- 管家层未就绪前，系统使用确定性规则链继续工作

## 当前 6 层架构

当前正式口径采用 6 层架构：

1. **交互与展示层**
   - Windows GUI、日志面板、模型选择、训练工作台、文档入口
2. **会话与运行时层**
   - 启动链、自检、安装包首启、固定管家层运行时状态、后台任务调度
3. **管家编排层**
   - `orchestrator`、意图识别、任务分发、多步计划、播报策略、事件摘要
4. **执行与知识层**
   - 用户可选执行层模型、知识库、专家结果解释、开放式回答
5. **通信与节点管理层**
   - `PC-Pi` 消息收发、多节点管理、事件回传、结果回传、ACK
6. **Pi 轻前端边缘层**
   - 唤醒、轻量 ASR、本地 TTS、本地停止播报、低频视觉前置检测、文本与关键帧上行

### 1. 语音闭环

1. Pi 本地离线识别音频为文本
2. 文本上行到 PC
3. PC 进入固定管家层编排
4. 再由执行层模型或专家执行链生成回答
5. 结果回传 Pi
6. Pi 本地播报

### 2. 视频闭环

1. Pi 低频前置检测
2. 危险事件或用户主动请求时上传关键帧
3. PC 进入固定管家层编排
4. 专家模型与知识库完成分析
5. 结果回传 Pi 并完成播报/ACK

### 3. 专家能力驱动调度

当前正式版本不再采用“某场景固定对应某专家”的硬编码主链。系统调度依据固定为：

1. 专家注册表中的结构化能力元数据
2. 专家知识域是否存在且可用
3. 当前输入类型
4. 当前上下文，包括节点来源、媒体类型、是否需要关键帧、是否为主动询问

固定管家层只允许在已注册专家中做选择，最终执行必须落回已注册的 `expert_code`。

## 当前正式边界

- 多节点目前以虚拟 Pi 和文件驱动链路验证为主
- 某些第三方语音扩展板存在兼容性风险，不纳入本版本正式通过项
- 真实麦克风阵列、真实扬声器阵列、实机集群长期联调仍需后续继续完成
- 固定管家层模型首次后台预热可能持续较长时间，但不应阻塞 GUI 显示与基本操作

## 当前验证结论

当前 `1.0.0` 已完成一轮围绕新安装链路的正式验收验证：

1. 真实安装包首启 smoke 已通过，GUI 先显示，固定管家层模型使用内置 GGUF 后台预热。
2. 固定管家层未就绪时，系统保持 `planner_backend=deterministic`，不会因模型未准备完成而阻塞主流程。
3. `PC` 本地语音、`Pi` 远端语音、`Pi` 边缘事件、本机摄像头巡检已统一进入 `orchestrator`。
4. 虚拟 Pi 与音频文件驱动链路已完成语音/视频闭环复验。
5. GUI 完整闭环与 GUI 发布验收闭环已通过，覆盖知识导入、专家资产导入、训练工作台、Ollama 模型选择、节点上线、事件回传与 ACK。
6. Ollama 模型选择页当前正式内置候选已确认包含：
   - `qwen3.5:4b`
   - `qwen3.5:9b`
   - `qwen3.5:27b`
   - `qwen3.5:35b`
